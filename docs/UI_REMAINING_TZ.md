# ТЗ: что осталось доделать по UI SubHub Admin Console

Дата аудита: 31 мая 2026  
Сайт для проверки: https://salkalin.github.io/my-subscriptions-app-v1-admin-dashboard/

Основа аудита: скриншоты экранов из задачи: дашборд, пользователи, подписки, аналитика сервисов, биллинг, маркетинг, геймификация, smart-аналитика.

## Цель

Довести текущий UI админ-панели SubHub из состояния красивого статического прототипа до полноценного рабочего интерфейса: все видимые кнопки, фильтры, вкладки, таблицы, графики и действия должны иметь понятное поведение, состояния загрузки/ошибок/пустых данных и пригодность для реальной работы оператора.

## Общий вывод

Визуальная база уже сильная: темная админка, единый сайдбар, карточки метрик, таблицы, бейджи, фильтры и CTA выглядят цельно. Основная оставшаяся UI-работа не в "нарисовать еще экранов", а в завершении интерактивности и рабочих сценариев.

Главные проблемы:

- Многие кнопки выглядят как рабочие, но по UI не видно результата: `Создать рассылку`, `Экспорт`, `Отчет PDF`, `Добавить сервис`, `Управление тарифами`, `Smart Action`, `Создать достижение`, `Назначить`, `Промокоды`, `Магазин наград`.
- Есть вкладки, которые должны открывать полноценные разделы, но выглядят как заготовки: биллинг, маркетинг, smart-аналитика, подписки.
- Таблицы показывают данные, но не хватает detail view, bulk actions, сортировки, выбора колонок, подтверждений и состояний.
- Графики местами выглядят декоративно или незавершенно, особенно экран биллинга: большой график "Выручка по дням" почти пустой.
- Нет видимых mobile/tablet состояний в переданных скринах; нужно отдельно проверить адаптив.
- Не хватает системных состояний: загрузка, ошибка API, пустой результат фильтра, нет прав, disabled/loading для кнопок.

## P0: обязательно до передачи как готовый UI

### 1. Опубликованный сайт и маршруты

Проверить, что сайт открывается по указанному URL и все разделы доступны после перезагрузки страницы.

Что сделать:

- Открыть `https://salkalin.github.io/my-subscriptions-app-v1-admin-dashboard/`.
- Проверить прямые URL для всех разделов:
  - `/dashboard`
  - `/users`
  - `/subscriptions`
  - `/analytics`
  - `/billing`
  - `/marketing`
  - `/gamification`
  - `/smart-analytics`
  - `/security`
  - `/settings`
- Если GitHub Pages отдает 404 на deep links, настроить SPA/Next export fallback или другой корректный способ публикации.

Критерий приемки:

- Любой раздел открывается напрямую из адресной строки и после refresh.
- Сайдбар подсвечивает активный раздел.
- Нет 404 GitHub Pages при прямом переходе.

### 2. Все видимые CTA должны работать

На каждом экране сейчас есть кнопки, которые выглядят как продуктовые действия. Их нельзя оставлять декоративными.

Минимум для каждого CTA:

- Открывается модалка, drawer, dropdown или отдельный экран.
- Есть `loading` при сохранении/экспорте.
- Есть success/error toast.
- Есть disabled-state, если действие недоступно.
- Если backend еще mock, в UI должно быть понятно, что действие выполнено локально/демонстрационно.

Обязательные CTA:

- Дашборд: `Отчет PDF`, `Создать рассылку`, `К деталям`, `Посмотреть список`.
- Пользователи: `Сегменты`, `Экспорт CSV`, `Создать рассылку`, иконка действия в строке пользователя, чекбоксы строк.
- Подписки: `Каталог сервисов`, `Добавить сервис`, строки таблицы, вкладка внешних подписок.
- Аналитика сервисов: `Сброс`, `Экспорт CSV`, фильтры.
- Биллинг: `Отчеты`, `Управление тарифами`, `Экспорт`, вкладки `Тарифы`, `Платежные шлюзы`, `Отчеты`.
- Маркетинг: `Промокоды`, `Создать рассылку`, `Сохранить`, `Превью`, `Отправить`, вкладки `Push`, `Промокоды`, `Реферальная программа`.
- Геймификация: `Магазин наград`, `Создать достижение`, `Назначить`, иконки редактирования пользователей.
- Smart-аналитика: `A/B-тесты`, `Smart Action`, `Сохранить`, `Создать сценарий`, переключатель Smart Actions.

Критерий приемки:

- У каждого CTA есть понятный результат клика.
- Нет кнопок-заглушек без реакции.

### 3. Таблицы пользователей, подписок и транзакций

Таблицы выглядят как рабочие, но сейчас не хватает основных operator-flow.

Что сделать:

- Добавить сортировку по ключевым колонкам.
- Добавить detail drawer по клику на строку.
- Добавить массовые действия после выбора строк.
- Добавить состояние `0 результатов` после фильтров.
- Добавить skeleton/loading.
- Добавить error-state с повтором запроса.
- Исправить пагинацию пользователей: на скрине показано `Показано 8 из 8`, но справа есть страницы `1 2 3 ... 2305`. Это конфликт данных.

Критерий приемки:

- Выбор чекбоксов открывает панель массовых действий.
- Клик по строке открывает подробности.
- Пагинация соответствует количеству данных.
- Фильтры, поиск и reset визуально меняют таблицу.

## P1: завершить рабочие сценарии по экранам

### Главный дашборд

Что видно:

- Экран хорошо собран визуально.
- Есть KPI, большой график, аномалии, топ сервисов, новые пользователи, риск оттока.

Что доделать:

- Сделать period switch `30д / 90д / 12м` реально интерактивным.
- Добавить tooltip на графики.
- Добавить оси/подписи там, где без них график читается как декоративная линия.
- Клик по аномалии должен открывать подробный drawer: причина, affected users/payments, рекомендуемые действия.
- `Посмотреть список` в риске оттока должен вести к отфильтрованному списку пользователей.
- `Создать рассылку` должен открывать создание кампании с предвыбранной аудиторией.
- `Отчет PDF` должен показывать процесс генерации и результат.

### Пользователи (CRM)

Что видно:

- Есть фильтры, сегменты, поиск, таблица, статусы, risk badges.

Что доделать:

- Detail drawer пользователя: профиль, подписки, платежи, коммуникации, достижения, события.
- Bulk actions: создать рассылку, экспорт выбранных, начислить баллы, заблокировать, сменить сегмент.
- Сегменты должны открывать менеджер сегментов, а не быть одиночной кнопкой.
- Иконка действия в строке должна иметь tooltip и понятное действие.
- Все `анонимизировано` допустимо для privacy-mode, но нужен переключатель/индикатор режима скрытия PII.
- Поиск должен показывать, по каким полям ищет: email, ID, телефон.

### Подписки

Что видно:

- Есть сводка внутренней подписки SubHub, вкладки, фильтры и таблица платящих.

Что доделать:

- Вкладка `Внешние подписки пользователей` должна иметь отдельную таблицу внешних сервисов.
- Строка подписки должна открывать detail drawer: тариф, история списаний, способ оплаты, продления, ошибки.
- Добавить действия: продлить, отменить, сменить тариф, заморозить, пересчитать.
- `Каталог сервисов` должен открывать каталог внешних сервисов с поиском, категориями и статусом интеграции.
- `Добавить сервис` должен открывать форму добавления сервиса.
- Добавить нормальное состояние для просроченных подписок и dunning-действий.

### Аналитика по сервисам

Что видно:

- Есть фильтры, KPI, топ сервисов, категории и кросс-срез возраст x категория.

Что доделать:

- Фильтры должны менять все блоки на экране и показывать активные chips.
- `Сброс` сейчас повторяется в шапке и внутри фильтров; оставить один понятный reset или развести смыслы.
- Добавить drilldown по сервису: кто подписан, выручка, churn, распределение по тарифам.
- Добавить tooltip к bar/progress-графикам.
- Экспорт CSV должен учитывать активные фильтры.

### Биллинг и финансы

Что видно:

- Есть вкладки, таблица последних транзакций и donut по платежным шлюзам.
- Большой график `Выручка по дням` визуально пустой.

Что доделать:

- Исправить график выручки: линия/столбцы, подписи, tooltip, легенда, пустое состояние.
- Реализовать вкладки `Тарифы`, `Платежные шлюзы`, `Отчеты`.
- Detail drawer транзакции: пользователь, шлюз, комиссия, статус, ошибка, refund.
- `Управление тарифами` должно открывать CRUD тарифов.
- `Отчеты` должны открывать выбор периода и формата.
- Добавить UI для возвратов, failed payments, повторных попыток списания.

### Маркетинг и коммуникации

Что видно:

- Есть форма email-рассылки, аудитория, отправка, preview, история кампаний, эффективность каналов, промокоды.

Что доделать:

- Заменить простой textarea на нормальный редактор/preview: subject, body, переменные, validation.
- `Превью` должно показывать письмо как получит пользователь.
- `Отправить` должно иметь confirm modal с количеством получателей.
- Добавить draft/scheduled/sent states.
- Вкладки `Push`, `Промокоды`, `Реферальная программа` должны быть полноценными экранами.
- Активные промокоды: добавить detail, создание, лимиты, срок действия, применимость к тарифам.

### Геймификация и достижения

Что видно:

- Есть KPI, каталог достижений, топ пользователей и эффект геймификации.

Что доделать:

- `Создать достижение`: форма с названием, условием, цветом/редкостью, наградой, лимитами.
- `Магазин наград`: отдельный экран с товарами, стоимостью, остатками, статусами.
- `Назначить`: выбор пользователя/сегмента и достижение/баллы.
- Карточки достижений должны иметь edit/detail state.
- Топ пользователей должен вести в профиль пользователя.

### Smart-аналитика и прогнозы

Что видно:

- Есть what-if sliders, прогноз, A/B-тесты, Smart Actions.

Что доделать:

- Слайдеры должны иметь числовые значения и reset.
- `Сохранить` должен сохранять сценарий и показывать его в списке.
- `Создать сценарий` должен открывать builder условия и действия.
- `Smart Action` должен открывать настройки автодействий.
- Вкладки `A/B-тесты`, `Когортный анализ`, `Прогнозы (ML)`, `Автосценарии` должны иметь полноценный UI.
- Для ML-прогнозов добавить confidence, model version, explainability/drivers.

## P2: системное качество UI

### Адаптив

Нужно отдельно проверить:

- 1440px desktop.
- 1024px tablet.
- 390px mobile.

Критерий приемки:

- Сайдбар превращается в drawer на мобильном.
- Таблицы не ломают экран: есть горизонтальный скролл, compact cards или column priority.
- Кнопки в шапке не вылезают за экран.
- Графики не сжимаются до нечитаемого состояния.

### Доступность и UX-мелочи

Что сделать:

- Tooltip для иконок без текста.
- Focus states для кнопок, ссылок, табов, фильтров.
- Keyboard navigation для табов, drawer, modal.
- `aria-label` для icon-only buttons.
- Контраст проверить на темной теме.
- Единый формат дат, валют и процентов.

### Единая дизайн-система

Что унифицировать:

- Высоты фильтров и search input.
- Стиль primary/secondary/destructive кнопок.
- Badge statuses: active/trial/overdue/risk/success/error.
- Empty/loading/error-компоненты.
- Drawer/modal headers, footers, confirm actions.

## Definition of Done

UI можно считать готовым, когда:

- Все экраны из сайдбара открываются напрямую и через навигацию.
- Все видимые кнопки, табы, фильтры и строки имеют рабочий сценарий.
- Для каждой таблицы есть search/filter/sort/detail/bulk или осознанное исключение.
- Для каждой async-операции есть loading, success и error.
- Нет декоративных графиков без данных, tooltip или пустого состояния.
- Проверены desktop/tablet/mobile.
- Добавлены/обновлены тесты на измененные интерактивные компоненты.

4. Исправить биллинг: график выручки и вкладки.
5. Завершить маркетинг: редактор, preview, отправка, промокоды.
6. Завершить подписки: detail/actions/catalog/add service.
7. Завершить smart-аналитику: сценарии, вкладки, сохранение.
8. Проверить адаптив и системные состояния.
